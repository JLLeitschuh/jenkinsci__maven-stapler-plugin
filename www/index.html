Servlet (and technologies like JSP/JSF that builds on top of it) can only bind request URLs into your static resources, JSPs, and/or servlets in a very limited way. You can even tell that from the java.net system. For example, the URL to view the archive of a mailing list is https://jaxb.dev.java.net/servlets/SummarizeList?listName=announce, the URL to search the archive if https://jaxb.dev.java.net/servlets/SearchList?listName=announce, and the URL to manage it is https://jaxb.dev.java.net/servlets/MailingListEdit?list=announce.

As you see, it is almost "procedual" --- operation names form the URL, and then the object to work with is given as a parameter. This has some problems.

  1. It prevents you from using the URL hierarchy to its original purpose of organizing your application.
  2. It makes your code procedual. All those servlets need to first identify your internal "MailingList" object first, by parsing the URL. This is even harder for JSPs. This is tedious and error-prone.
  3. It makes relative URLs less useful, because most of the time whenever you generate a hyperlink, you have to do that by something like /servlets/MailingListEdit?list=${listName}&project=${projectName}

The Stapler project considers that these problems are caused by a lack of intuitive bindings between URLs and servlets/JSPs. To solve this problem, Stapler binds URLs to your model objects directly, invoking methods or JSPs for that object.

The following list describes this process in more details:

  1. Your web application installs the Stapler servlet as the default servlet, and it will take over the way requests are mapped to servlets/JSPS/resources.
  2. You write model objects to represent your web application. For example, in the java.net system example, you'll write a JavaNet class for the root, Project class, then MailingList class.
  3. Your web application registers the root object (the sole instance of JavaNet) to the Stapler at start-up.
  4. Stapler maps the incoming URLs to one of your model objects. For example, if the URL is /project/jaxb/mailingList/foo/archive, then app.getProject("jaxb").getMailingList().get("foo").doArchive(request,response) (depending on your object model) is evaluated and this method will process the request.
  5. JSPs and static resources are served in a similar fashion.
